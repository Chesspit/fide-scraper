# 2026-04-20 — VPS CPU spike + backfill crash

## Timeline

- **14:18–14:49 lokal** — Hostinger VPS CPU steigt von ~10 % auf 99 %.
  Aktiver Prozess laut Hostinger: `postgres: fide fidedb (local) SELECT`.
- **~16:00 UTC** — 2014-Backfill in tmux-Session `backfill2014` gestartet
  (`docker compose -f docker-compose.vps.yml run --rm scraper python -m scripts.backfill`).
- **18:33 UTC** — Backfill bricht ab:
  `psycopg2.OperationalError: server closed the connection unexpectedly`.
  Die DB selbst bleibt stabil (Uptime 11 Tage, keine Restarts).
- **danach** — Loop ruft für 2,5 h weiter `save_period()` auf, jeder Call wirft
  `InterfaceError: connection already closed`. tmux wird gekillt und der
  Backfill neu gestartet; `get_pending_periods()` überspringt die bereits
  erledigten 3.375 Kombinationen verlustfrei.

## Root cause (CPU-Spike)

`scripts/resolve_opponents.py::build_rating_history_index(conn, fide_ids)` führt
aus:

```sql
SELECT fide_id, period, published_rating
FROM rating_history
WHERE published_rating IS NOT NULL
  AND fide_id = ANY(%s)
```

Im Fuzzy-Modus erweitert sich `relevant_fids` auf sämtliche FIDE-IDs aus
`by_name` (~1,78 Mio.). Postgres muss diesen Array gegen die 4,8 Mio. Zeilen
grosse `rating_history` joinen → Hash-Join mit massivem CPU-Bedarf.

Während der Resolver-v4-Entwicklung lief das 6× zwischen 12:00–16:00 UTC
(Threshold- + Anchor-Iteration), was die Last kumuliert hat.

Gemini hatte `IN (SELECT ...)` + fehlende Indizes diagnostiziert; beides trifft
nicht zu (`fide_id` ist PRIMARY KEY auf `players` und `rating_history`; die
fragliche Query steht nirgends im Code). Der Backfill war nicht die Ursache —
er wurde erst nach dem Spike gestartet.

## Secondary issue (Backfill-Crash)

`scraper/db.py::save_period` besitzt kein Reconnect/Retry. Ein einzelner
transient gedroppter Connect killt den gesamten Lauf, obwohl
`get_pending_periods()` einen sauberen Resume erlauben würde.

## Planned fixes

1. **`build_rating_history_index`**: bei `len(fide_ids) > 500_000` den
   `ANY(%s)`-Filter weglassen und stattdessen einen Full-Scan mit
   Python-seitiger Filterung fahren. Vermeidet, dass Postgres einen
   Millionen-Element-Array deserialisieren und hashen muss.
2. **`statement_timeout`** (z. B. 15 min) per Session-Setting in
   `scraper/db.py`, `scripts/resolve_opponents.py` und `scripts/backfill.py`
   setzen — Sicherheitsnetz gegen Runaway-Queries.
3. **Reconnect-Wrapper** um `save_period` in `scraper/db.py`. Verschoben, bis
   der aktuell laufende 2014-Backfill durch ist, um Race Conditions mit
   offenen Cursoren zu vermeiden.

## Wie diagnostizieren bei Wiederholung

- `tmux ls` auf dem VPS prüfen, `tail /opt/fide-scraper/backfill_*.log` und
  `resolve_*.log` inspizieren, bevor DB-Schaden vermutet wird.
- Die DB selbst war in diesem Vorfall durchgehend gesund; alle Symptome
  lagen auf Client-/Query-Ebene.
